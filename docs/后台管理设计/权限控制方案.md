# 后台管理系统权限控制方案

## 1. 设计概述

### 1.1 设计目标
为AI工作流平台后台管理系统设计一套完整、灵活、安全的权限控制体系，实现：
- **精细化权限控制**：基于角色（RBAC）的权限分配，支持细粒度操作控制
- **操作安全审计**：完整记录所有管理员操作，便于追溯和审计
- **数据权限隔离**：不同角色只能访问和管理分管的数据范围
- **动态权限调整**：支持权限的动态分配和回收，无需修改代码

### 1.2 核心原则
1. **最小权限原则**：每个角色只拥有完成其职责所必需的最小权限
2. **职责分离原则**：敏感操作需要多角色协作完成，防止单点风险
3. **操作可追溯原则**：所有操作必须有完整日志记录，支持事后审计
4. **权限时效性原则**：支持临时权限和权限过期机制

## 2. RBAC模型设计

### 2.1 核心实体关系
```
管理员 (AdminUser) --belongs-to--> 角色 (Role)
角色 (Role) --has-many--> 权限 (Permission)
权限 (Permission) --hierarchical--> 父权限
操作日志 (OperationLog) --belongs-to--> 管理员
```

### 2.2 实体详细定义

#### 2.2.1 管理员 (AdminUser)
| 字段 | 说明 | 约束 |
|------|------|------|
| id | 唯一标识 | 主键 |
| username | 登录用户名 | 唯一，不可变 |
| password_hash | 密码哈希 | BCrypt算法 |
| role_id | 所属角色ID | 外键约束 |
| status | 账户状态 | 正常/禁用/锁定 |
| last_login_at | 最后登录时间 | |
| login_fail_count | 连续登录失败次数 | 达到阈值锁定 |

#### 2.2.2 角色 (Role)
| 字段 | 说明 | 示例值 |
|------|------|--------|
| id | 唯一标识 | 1 |
| role_name | 角色名称 | 超级管理员 |
| role_code | 角色编码 | super_admin |
| is_system | 是否系统内置 | 1（是） |
| description | 角色描述 | 系统最高权限管理者 |

#### 2.2.3 权限 (Permission)
| 字段 | 说明 | 示例值 |
|------|------|--------|
| id | 唯一标识 | 1 |
| permission_code | 权限编码 | workflow:list |
| module | 所属模块 | workflow |
| action | 操作类型 | read |
| parent_id | 父权限ID | 0（根权限） |
| sort_order | 排序权重 | 10 |

#### 2.2.4 权限编码规范
```
格式：{模块}:{操作}

模块分类：
- workflow: 工作流管理
- user: 用户管理  
- task: 任务管理
- order: 订单管理
- system: 系统管理
- admin: 管理员管理

操作类型：
- create: 创建
- read: 查看
- update: 编辑
- delete: 删除
- publish: 发布/上下架
- export: 导出
- audit: 审核
- statistics: 统计
```

## 4. 权限验证流程

### 4.1 前端权限控制
```javascript
// 路由守卫权限检查
router.beforeEach(async (to, from, next) => {
  // 1. 获取当前用户权限列表
  const permissionStore = usePermissionStore();
  await permissionStore.loadPermissions();
  
  // 2. 检查路由权限要求
  const requiredPermission = to.meta?.permission;
  if (requiredPermission) {
    const hasPermission = permissionStore.hasPermission(requiredPermission);
    if (!hasPermission) {
      // 跳转403页面或返回上一页
      next('/403');
      return;
    }
  }
  
  // 3. 动态菜单生成
  const hasAccess = permissionStore.checkRouteAccess(to);
  if (!hasAccess) {
    next('/404');
    return;
  }
  
  next();
});
```

### 4.2 后端权限拦截
```java
// Spring Security权限拦截器
@Component
public class PermissionInterceptor implements HandlerInterceptor {
    
    @Override
    public boolean preHandle(HttpServletRequest request, 
                            HttpServletResponse response, 
                            Object handler) throws Exception {
        
        // 1. 获取当前管理员信息
        Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
        AdminUserDetails adminDetails = (AdminUserDetails) authentication.getPrincipal();
        
        // 2. 获取请求路径和方法
        String requestURI = request.getRequestURI();
        String httpMethod = request.getMethod();
        
        // 3. 映射到权限编码
        String permissionCode = PermissionMapping.getPermissionCode(requestURI, httpMethod);
        
        // 4. 验证权限
        if (!adminDetails.hasPermission(permissionCode)) {
            response.setStatus(HttpStatus.FORBIDDEN.value());
            response.getWriter().write("权限不足");
            return false;
        }
        
        return true;
    }
}
```

## 5. 数据权限设计

### 5.1 数据隔离级别
| 隔离级别 | 描述 | 适用场景 |
|---------|------|---------|
| 全局访问 | 可访问所有数据 | 超级管理员 |
| 部门隔离 | 只能访问所在部门数据 | 大企业多部门场景 |
| 个人隔离 | 只能访问自己创建的数据 | 工作流管理员 |
| 混合模式 | 部分数据全局，部分个人 | 通用后台管理系统 |

### 5.2 本系统数据权限方案
采用**个人数据隔离+全局只读**的混合模式：

1. **工作流数据**
   - 创建者：拥有完整权限（CRUD）
   - 其他管理员：只读权限（read）
   - 超级管理员：所有权限

2. **用户数据**
   - 用户管理员：完整权限
   - 其他管理员：只读权限（read）
   - 超级管理员：所有权限

3. **任务数据**
   - 任务所属用户：完整权限
   - 工作流管理员：查看关联工作流的任务
   - 超级管理员：所有权限

### 5.3 数据权限SQL实现
```sql
-- 工作流列表查询（带数据权限）
SELECT * FROM workflows w
WHERE 
  -- 超级管理员无限制
  (:currentAdminRole = 'super_admin')
  OR
  -- 工作流管理员查看自己创建的工作流
  (:currentAdminRole = 'workflow_manager' AND w.creator_id = :currentAdminId)
  OR
  -- 其他角色只能查看已上架的工作流
  (w.status = 1)
ORDER BY w.created_at DESC
LIMIT :pageSize OFFSET :offset;
```

## 6. 操作审计系统

### 6.1 审计日志记录范围
必须记录的操作：
- 所有登录/登出操作
- 所有增删改操作（CUD）
- 敏感查询操作（如查看用户敏感信息）
- 系统配置变更
- 权限分配变更

可选记录的操作：
- 普通查询操作（可配置开关）
- 导出操作（记录导出条件和数量）

### 6.2 审计日志字段设计
| 字段 | 类型 | 说明 |
|------|------|------|
| admin_id | BIGINT | 操作管理员ID |
| admin_username | VARCHAR | 管理员用户名（冗余） |
| module | VARCHAR | 操作模块 |
| action | VARCHAR | 操作类型 |
| action_name | VARCHAR | 操作描述 |
| target_id | VARCHAR | 操作目标ID |
| target_type | VARCHAR | 操作目标类型 |
| request_method | VARCHAR | HTTP方法 |
| request_url | VARCHAR | 请求URL |
| request_params | JSON | 请求参数 |
| ip_address | VARCHAR | 客户端IP |
| user_agent | VARCHAR | 用户代理 |
| status | TINYINT | 操作状态（成功/失败） |
| error_message | TEXT | 错误信息（失败时记录） |
| execution_time | INT | 执行时间（毫秒） |
| created_at | DATETIME | 操作时间 |

### 6.3 审计日志存储策略
1. **实时存储**：所有操作立即写入数据库
2. **分区存储**：按月分区，便于管理和查询
3. **归档策略**：超过180天的数据归档到历史表
4. **备份策略**：每日备份，保留30天

### 6.4 审计日志查询功能
支持以下查询条件：
- 操作时间范围
- 操作管理员
- 操作模块
- 操作类型
- 操作状态（成功/失败）
- 目标ID/类型

## 7. 安全增强措施

### 7.1 权限变更审计
所有权限分配变更必须记录：
1. **角色权限变更**：记录变更前后的权限列表
2. **管理员角色变更**：记录变更前后的角色
3. **敏感操作**：需要二次确认或多人审批

### 7.2 操作风险控制
1. **批量操作限制**：批量删除、状态变更等操作限制数量
2. **敏感操作确认**：重要数据删除需要二次确认
3. **操作频率限制**：防止恶意刷操作

### 7.3 数据备份与恢复
1. **定期备份**：每日全量备份+实时增量备份
2. **备份验证**：定期验证备份数据完整性
3. **恢复演练**：每季度进行数据恢复演练

## 10. 风险与应对

### 10.1 技术风险
| 风险点 | 可能性 | 影响 | 应对措施 |
|--------|--------|------|---------|
| 权限设计过于复杂 | 中 | 高 | 采用RBAC标准模型，保持简洁 |
| 性能影响显著 | 低 | 中 | 优化SQL查询，添加缓存 |
| 数据权限实现困难 | 中 | 中 | 分阶段实施，先实现基础权限 |

### 10.2 管理风险
| 风险点 | 可能性 | 影响 | 应对措施 |
|--------|--------|------|---------|
| 权限分配混乱 | 高 | 中 | 制定权限分配规范，定期审计 |
| 操作日志数据量过大 | 中 | 低 | 实施分区存储和定期归档 |
| 安全漏洞利用 | 低 | 高 | 定期安全扫描，及时修复漏洞 |
