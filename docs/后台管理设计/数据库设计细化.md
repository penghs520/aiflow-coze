# 后台管理系统数据库设计细化

## 1. 设计概述

基于核心业务数据库（用户、工作流、任务、订单、资源点记录），后台管理系统需要以下扩展表支持：

1. **管理员表**：后台管理员账户信息
2. **角色表**：角色定义与权限分组
3. **权限表**：具体操作权限定义
4. **操作日志表**：管理员操作审计
5. **系统配置表**：平台运行参数配置
6. **通知公告表**：系统公告与用户通知

## 2. 新增表结构设计

## 3. 表关系图

### 3.1 核心关系
1. **管理员 ↔ 角色**：多对一关系，一个管理员属于一个角色，一个角色有多个管理员
2. **角色 ↔ 权限**：多对多关系，通过`admin_role_permissions`关联表连接
3. **操作日志 ↔ 管理员**：多对一关系，一个操作日志属于一个管理员
4. **通知公告 ↔ 管理员**：多对一关系，一个公告由一个管理员创建

### 3.2 数据流设计
- 管理员登录后，系统根据其角色加载对应的权限列表
- 管理员操作时，权限拦截器验证`permission_code`是否在权限列表中
- 所有操作自动记录到操作日志表，包括成功和失败的操作
- 系统配置支持热更新，无需重启应用

## 5. 数据库优化建议

### 5.1 索引优化
1. **操作日志表**：按时间范围查询频繁，建立`created_at`降序索引
2. **管理员表**：登录查询使用`username`唯一索引，结合状态查询
3. **角色权限表**：关联查询使用组合索引`(role_id, permission_id)`

### 5.2 分区策略
- **操作日志表**：按月分区，便于历史数据归档和删除
- **通知公告表**：按年份分区，过期数据自动归档

### 5.3 读写分离
- 管理员操作（写）：主库
- 数据查询（读）：从库，特别是操作日志和大数据量查询

### 5.4 数据归档
- 操作日志保留180天，过期数据转移到归档表
- 通知公告过期后标记为历史状态，可查询但不可编辑

## 6. 安全设计

### 6.1 密码安全
- 使用BCrypt算法存储密码哈希
- 强制密码复杂度：至少8位，包含字母、数字和特殊字符
- 密码错误次数限制：连续5次失败锁定账户30分钟

### 6.2 操作审计
- 所有增删改操作记录完整日志，包括操作前后数据快照
- 敏感操作（用户状态修改、资源点调整）需要二次确认
- 操作日志不可删除，仅支持归档

### 6.3 数据权限
- 基于角色的数据权限控制：不同角色只能看到分管的数据
- 工作流管理员只能管理工作流，不能操作用户数据
- 财务管理员只能查看订单和财务数据，不能修改系统配置
